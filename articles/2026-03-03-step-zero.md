# Before You Guard It, Inventory It: The Case for Step Zero

Here's how most quality processes get designed:

1. What could go wrong?
2. What should we check?
3. How do we enforce the checks?

It's a reasonable sequence. Identify the risks. Design the mitigations. Wire them in. Except it skips a question that turns out to be more important than all three:

**What do we actually have?**

## The phantom artifact problem

Imagine designing a quality checklist for a codebase. You include: "Verify that documentation matches implementation." Sounds like a good check.

Except there is no documentation.

This isn't a hypothetical. It happens constantly. Quality processes reference artifacts that don't exist — architecture docs, testing strategies, API specs, decision logs, onboarding guides. The check passes vacuously (you can't find a mismatch between code and docs if there are no docs) or it gets skipped ("not applicable"), and the real finding — *this system can't explain itself* — is never surfaced.

The phantom artifact problem is insidious because it looks like a passing check. "Docs match implementation? Well, there are no docs, so... technically no mismatch. Check."

## Step Zero

The fix is simple: before you analyze what could go wrong, catalogue what actually exists. We call this Step Zero — the system inventory.

Step Zero asks three kinds of questions:

**What's present?** Not "what should exist" but "what does exist." Does this codebase have an architecture doc? Does this API have a spec? Does this project have a testing strategy document? Does this documentation system have a stated purpose?

**What's the state of what's present?** Not just "does it exist" but "when was it last updated? Does it reference current reality or a past version? Is anyone actually using it?" An architecture doc from 2019 is present but potentially worse than absent — it misleads.

**What's the gap classification?** This is where it gets interesting. For each expected artifact that's missing, you need to decide:

- **Missing and needed** — this system should have this artifact. The quality process should include a bootstrap action: "create a 2-sentence intent statement before proceeding."
- **Missing and fine** — this system legitimately doesn't need this. A small CLI tool doesn't need an API spec. A personal project doesn't need an onboarding guide. Not every system needs every artifact.
- **Present but stale** — this exists but shows signs of age. Dates that are years old. References to removed features. Links that 404. This is often worse than "missing" because it creates false confidence.

The "missing and fine" classification is crucial. Without it, inventories become wishlists — everything that *could* exist gets flagged as missing, and the resulting quality process drowns in checks for artifacts that would add no value to this specific system. A good inventory distinguishes absence-as-problem from absence-as-appropriate.

## The highest-value finding

When you run Step Zero against a real system, the most impactful finding is almost never about what's wrong with existing artifacts. It's about what *doesn't exist that should*.

The most fundamental version: **does this system have a stated intent?** Can you find, somewhere in the system's artifacts, a clear statement of what it's for and why it exists?

If not, every subsequent quality check is ungrounded. You can check consistency (do the parts agree with each other?) and you can check mechanical correctness (do the links work?). But you can't check alignment — you don't know what the system is supposed to be aligned *with*.

This is why Step Zero matters more than the risk analysis that follows it. A system with no intent documentation and perfect internal consistency is more fragile than a system with clear intent and a few broken links. The links are cheap to fix. The intent, once lost, is expensive to recover — if it can be recovered at all.

## How Step Zero changes the output

When we added Step Zero to our guard generators, the guards they produced changed materially.

**Before Step Zero**, the generator would produce checks like:
- "Does documentation match implementation?"
- "Are architectural decisions recorded?"
- "Does the test suite reflect the current contract?"

These checks assume the referenced artifacts exist. If they don't, the check is meaningless.

**After Step Zero**, the generator produces guards with two layers:

First, **bootstrap checks** — one-time actions to create prerequisites that don't exist yet:
- "Write a 2-sentence intent statement."
- "Document the top 3 architectural conventions."
- "Generate an API spec from the current implementation."
- "Define the standard error format."

Then, **recurring checks** — the ongoing quality process, which can now reference artifacts that actually exist (or that the bootstrap step created).

This two-layer structure means the guard works for systems at any maturity level. A greenfield project gets bootstrap actions. A mature project gets straight to the recurring checks. The same generator, the same process — Step Zero adapts the output to the system's actual state.

## Applying this beyond quality processes

The Step Zero pattern isn't limited to quality processes. It applies to any situation where you're designing an intervention for a system you don't fully understand yet:

**Consulting engagement?** Before recommending process improvements, inventory what processes actually exist and which are followed versus documented-but-ignored.

**New team member onboarding?** Before reading the onboarding guide, check whether the onboarding guide exists, when it was last updated, and whether anyone has used it recently.

**Architecture review?** Before evaluating the architecture, check whether there are architecture docs, ADRs, or any record of why the current structure exists.

**Migration planning?** Before planning the migration, inventory what actually needs to migrate — not what the docs say exists, but what a scan of the actual system reveals.

In each case, the inventory surfaces the most important finding early: the gap between what the system *claims* about itself and what's actually there. That gap is, in a very real sense, the system's entropy level. A low-entropy system can accurately describe itself. A high-entropy system requires archaeology.

## The cost of skipping it

Quality processes designed without an inventory tend to have a specific pathology: they check for the *wrong things*. Not wrong in the sense of unimportant, but wrong in the sense of *not applicable to this specific system*.

A generic code review checklist applied to a documentation project wastes time on code-specific checks and misses documentation-specific risks. A testing strategy template applied to a system with no tests produces a strategy for a test suite that doesn't exist. A documentation quality process applied to a codebase with excellent auto-generated API docs wastes time checking things that are already mechanically guaranteed.

Step Zero prevents this. By cataloguing what exists before analyzing what could go wrong, you ensure the quality process is shaped for the system it's actually guarding — not for a hypothetical system that has all the artifacts your template assumes.

Start with what you have. Then decide what to check.
